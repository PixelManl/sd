# 初赛详细执行指南 README

## 1. 文档定位

这份文档是在 [初赛技术路线README.md](E:\Repository\Project\sd\docs\初赛技术路线README.md) 的基础上，进一步回答三个执行问题：

1. 检测具体用什么模型
2. 生成线具体怎么利用现有资源
3. 剩余 7 天内，仓库、服务器、人力、提交物如何组织

这份文档的判断来自本地规范 + 当前仓库 + 多智能体只读评审结果，不依赖外网。

## 2. 多智能体评审结论

本轮共使用 3 个子代理并行审查：

- 检测选型代理：聚焦 `bolt/detect/`，结论是主推 `Ultralytics YOLO11n`，备选 `YOLO11s`
- 生成线代理：聚焦 `bolt/generate/`，结论是生成线只能降级成“最小可运行提交物 + 少量高置信补样工具”
- 提交组织代理：聚焦 `bolt/package/` 和比赛格式，结论是接下来 7 天重心应是“冻结、打包、文档、上传彩排”

三路意见高度一致：

- 不再扩实验面
- 不再把生图质量打磨当主胜负手
- 检测先卡位
- 数据集冻结优先
- 生图做最小包
- 提交组织必须前置

## 3. 检测线执行指南

### 3.1 主推模型

- 主推：`Ultralytics YOLO11n`
- 备选：`Ultralytics YOLO11s`

### 3.2 为什么这么选

原因不是“理论最强”，而是“7 天内最容易闭环”。

- 当前仓库的检测骨架默认后端就是 `ultralytics`
- 默认配置已经写了 `yolo11n.pt`
- 当前阶段是单类 `missing_fastener`
- 当前路线要求“先上最轻 baseline，不做检测框架横跳”

规范与仓库依据：

> `初赛技术路线README` 已明确写了“只做单类、先上最轻 baseline、不做检测框架横跳”。  
> `baseline.yaml` 的默认后端为 `ultralytics`，默认权重为 `yolo11n.pt`。  
> 来源：`docs/初赛技术路线README.md`，`bolt/detect/configs/baseline.yaml`

### 3.3 不推荐的做法

以下内容现在不要做：

- 不要切到 MMDetection、Detectron2、RT-DETR 等新框架
- 不要在 `YOLO11n / YOLO11s / YOLO11m` 之间来回扫
- 不要先调模型，再补数据合同
- 不要用随机 split 顶替场景隔离 split

### 3.4 当前真实缺口

当前检测线离“实跑”还差这几步：

1. 缺陷真值数据尚未接成唯一 detector-ready 数据入口
2. `prepare_detection_dataset.py` 对 COCO 分支仍偏摘要，真正可落成 YOLO 训练集的是 VOC 风格分支
3. `pyproject.toml` 还没有显式声明 `ultralytics`
4. 数据切分仍需要从“随机”升级成“scene/capture_group 防泄漏切分”
5. 单测主要验证 CLI 和 mock，不代表真实训练已跑通

### 3.5 检测线最小落地顺序

#### Day 1-2

- 冻结唯一任务边界：只做 `missing_fastener`
- 整理 100 张真实缺陷图和对应框标注
- 为每个样本补齐最少元信息：
  - `sample_id`
  - `scene_id`
  - `capture_group_id`
- 先做场景隔离切分，再做 train/val/test

#### Day 3

- 产出真正 detector-ready 的 YOLO 数据集：
  - `images/train`
  - `images/val`
  - `images/test`
  - `labels/train`
  - `labels/val`
  - `labels/test`
  - `dataset.yaml`

#### Day 4

- 跑 `YOLO11n` 第 1 轮
- 固定 `imgsz=1024`
- 拿到第一份 val 指标
- 导出误检 / 漏检样例

#### Day 5

- 只修数据问题和阈值问题
- 跑第 2 轮
- 如果召回明显不足，再切 `YOLO11s`
- 不换框架，不改大结构

#### Day 6-7

- 冻结 best 权重
- 跑评估和文件夹推理
- 导出：
  - `metrics.json`
  - `predictions.json`
  - 误检 / 漏检示例
- 补齐检测最小运行说明

### 3.6 检测线的 Done Definition

- 有一条真正跑通的训练链
- 有一份固定的 val 结果
- 有可视化误检 / 漏检
- 有 best 权重
- 有最小推理入口

## 4. 生成线执行指南

### 4.1 生成线角色重定义

生成线现在不是主线，不再负责“证明方法先进”，只负责两件事：

1. 形成一个最小可运行生图模型提交物
2. 产出少量高置信生成样本，作为训练补样工具

### 4.2 只保留哪条生成链

只保留：

- `[run_sdxl_oriented_batch.py](E:\Repository\Project\sd\bolt\generate\scripts\run_sdxl_oriented_batch.py)`

只允许把它当：

- 单配置
- 小批量
- 可复现
- 可筛选

的最小提交线。

### 4.3 现在必须停掉的内容

从现在起全部停掉：

1. `缺陷 -> 健康`
2. 双向路线
3. 大规模 prompt / ROI / mask / geometry prior 搜索
4. 全量 SAM2 资产化
5. queue 式长任务生图实验
6. donor copy-paste / IP-Adapter / distance ladder 作为主线继续扩

这些内容可以保留为历史实验，但不能再吃主资源。

### 4.4 现有 SAM2 / mask 资产怎么用

正确用法只有三种：

1. 控编辑范围
2. 做人工质检
3. 保留数据谱系

具体对应：

- `core_mask`：贴紧缺陷证据，用于裁 ROI
- `edit_mask`：给 inpainting 一个略大编辑区
- `overlay`：供人工复核
- `metadata / manifest`：记录 lineage、QA 状态、seed、参数

错误用法：

- 不要把 mask 当官方标注替代物
- 不要把 SAM2 draft 资产直接入最终数据集

### 4.5 什么样的生成样本可以入最终数据集

必须同时满足：

1. 仍然属于 `missing_fastener`
2. 局部变化成立，但整体结构、光照、背景、金具几何不漂
3. 没有多螺母、悬浮件、套筒、熔融金属、长尾巴、假零件
4. 只进入训练集，不进入 val/test
5. 保留来源图、seed、参数、QA 记录
6. 通过人工复核

建议策略：

- 宁可只收 `0-20` 张精筛样本，也不要批量灌库

### 4.6 生成线门控策略

建议两段式：

#### 第一段：门控批

- 先跑 `5-10` 张
- 只看可用率

如果人工可用率 `< 30%`：

- 当天停止扩样
- 生图降级为“提交展示包”
- 不再作为数据增广来源

#### 第二段：生产批

- 只有门控批通过后，才跑 `20-30` 张
- 依旧人工筛图

### 4.7 生成线资源分配

生成线内部建议这样分：

- `70%` 人力：白名单样本选择、mask QA、人工验收
- `20%` GPU：两批推理
- `10%` 资源：一次性诊断实验

不要把服务器连续挂满去跑整套 queue。

## 5. 数据集冻结指南

### 5.1 数据集主体原则

最终提交数据集必须以真实数据为主体。

建议：

- 主体：100 张真实缺陷图 + 真实框标注
- 补充：少量人工复核通过的生成样本

### 5.2 冻结时必须做的事

1. 清洗错误标注
2. 剔除歧义样本
3. 统一类别名
4. 统一文件命名
5. 冻结 split
6. 统计样本规模、类别分布、场景分布
7. 明确局限性

### 5.3 特别注意

标注边界必须遵守官方说明：

- 只支持矩形框
- 被遮挡或视角不清时不标
- 某些位置本身无螺母，不应误标

## 6. 提交与打包指南

### 6.1 对外最终提交物

只保留三件：

1. `团队名_生图模型成果物.zip`
2. `团队名_高质量数据集成果物.zip`
3. `团队名_高质量数据集说明文档.docx`

### 6.2 本地 staging 目录建议

建议本地先构造：

```text
bolt/package/build/2026-03-31-final/
├─ model_pkg/
├─ dataset_pkg/
├─ docs/
└─ checks/
```

对外始终只导出三件最终文件。

### 6.3 生图模型包最小结构

```text
model_pkg/
├─ model/
│  └─ core/
├─ code/
│  └─ infer.py
├─ env/
│  ├─ requirements.txt
│  └─ env_note.md
├─ ascend_image/
│  └─ ascend910b_model_env.tar
├─ docs/
│  └─ 模型运行手册_v1.0.pdf
└─ README.md
```

### 6.4 高质量数据集包最小结构

```text
dataset_pkg/
├─ DataFiles/
└─ Annotations/
```

不要混入：

- mask
- overlay
- review 截图
- split 可视化
- 日志
- 私有路径

### 6.5 说明文档最容易漏的项

最容易漏掉的执行字段：

1. 访问渠道
2. 特殊业务规则
3. 局限性说明
4. 版本控制记录
5. 安全规范
6. 使用许可
7. 评估方法
8. 典型应用案例

### 6.6 平台上传最容易错的地方

1. 生图模型上传到“我的模型”，不是样本集
2. 说明文档挂在样本集介绍页“附件”
3. 数据集本体走“数据预览 -> 添加文件”

## 7. 总资源分配建议

从整体晋级概率出发，建议按下面分配：

- `50%`：真实数据冻结 + 检测提交闭环
- `20%`：说明文档 + 打包 + 上传彩排
- `20%`：最小可运行生图包
- `10%`：SAM2 重点资产 + 生成样本复核

这 7 天不按“哪个实验更有趣”分资源，只按“哪个动作更接近最终提交物”分资源。
